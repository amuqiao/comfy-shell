一文读懂 ComfyUI

一个把 AI 生成过程拆成节点、连成图的开源引擎:每个节点是一步计算,每根连线是一份数据,整张图就是一条可保存、可复现、可编程的生成流水线。

Python · PyTorchGPL-3.080k+ Stars节点图 = 工作流图像 / 视频 / 音频
workflow: text2img.json 点击 Queue Prompt,看一次真实的执行顺序 →
MODEL CLIP VAE CONDITIONING LATENT IMAGE
这就是 ComfyUI 里最经典的「文生图」工作流 —— 连线颜色 = 数据类型,类型对得上才允许连接。

它是什么,解决什么问题

POSITIONING

ComfyUI 是一个开源的节点式(node-based)AI 生成工作流引擎,最初由 comfyanonymous 于 2023 年初发布,现由 Comfy Org 维护。它让你用「搭积木连线」的方式,自由组合 Stable Diffusion、Flux、SD3、各类视频/音频模型的每一个计算环节。

它要解决的核心问题是:传统 WebUI 把生成流程写死成一张固定表单——你只能填参数,不能改流程。而扩散模型的真实流程(编码 → 采样 → 解码)本质上是一条可任意重组的数据流水线。ComfyUI 把这条流水线「掀开」给你:每一步是节点,步骤间的数据是连线,想怎么组装就怎么组装,并且整张图可以存成 JSON、嵌进 PNG、通过 API 复现。

✔ ComfyUI 是

  • 可视化工作流编辑器 + 后端执行引擎,两者可分离使用
  • 本地优先、离线可跑的推理框架(也被云平台大量托管)
  • 显存管理极强的运行时:8GB 甚至更低显存也能跑大模型(智能分块加载/卸载)
  • 庞大生态的宿主:数千个社区 custom nodes 扩展一切能力
  • 可编程后端:任何程序都能 POST /prompt 提交工作流

✘ ComfyUI 不是

  • 不是模型本身——它不训练模型,只加载与推理已有权重
  • 不是「一键出图」傻瓜工具——上手门槛高于 WebUI,换来的是自由度
  • 不是通用工作流编排平台(如 Airflow)——它专为生成式模型的张量数据流设计
  • 不只是画图界面——前端只是壳,核心价值在执行引擎与模型运行时

一句话总结:ComfyUI = 节点图前端 + DAG 执行引擎 + 扩散模型运行时。理解了这三层,就理解了整个项目。下面顺着一张图的「一生」讲清楚。

核心数据流:一张图是怎么被算出来的

LATENT DIFFUSION

ComfyUI 的节点不是随意发明的,它们一一对应潜空间扩散模型(Latent Diffusion)的真实计算阶段。首屏那 7 个节点,其实就是下面这条流水线。点击每个阶段查看细节:

亲手感受「去噪」:扩散模型的本质

所谓「生成」,就是从纯噪声出发,让模型一步步预测并减去噪声,直到浮现出与文字描述匹配的图像。拖动滑块模拟 KSampler 的采样步:

每一步,采样器(euler、dpm++ 2m 等)按调度器给定的噪声水平,调用扩散模型预测噪声并从潜空间中移除。步数越多细节越收敛,但耗时线性增长——这就是 KSampler 节点上 steps / sampler_name / scheduler 三个参数的含义。

关键洞察:这四个阶段之间流动的数据类型各不相同(文本 → CONDITIONING,噪声 → LATENT,图像 → IMAGE)。ComfyUI 给每种中间数据定义了强类型端口,连线即类型检查——这就是为什么它能安全地让你重排流程:在采样前插入 ControlNet 注入构图、在 MODEL 上挂 LoRA 微调风格、把 VAE Decode 换成放大再解码……都是合法的图变换。

执行原理:图如何变成计算

EXECUTION ENGINE

点下 Queue Prompt 后发生了什么?前端把整张图序列化成一份 JSON(节点 id、类型、参数、连接关系),POST /prompt 发给后端;后端校验后放入队列(PromptQueue),由执行器(execution.py)逐个消费:

1. 反向求值的拓扑执行 —— 执行器从输出节点(如 Save Image)出发,沿连线反向解析依赖,对 DAG 做拓扑排序,保证每个节点在其所有输入就绪后才执行。没有被任何输出依赖的「孤岛节点」根本不会运行——这是惰性求值。

2. 结果缓存(增量执行) —— 每个节点的输出会以「节点参数 + 上游输入」为键缓存。再次排队时,只有发生变化的节点及其下游会重新计算。改个提示词,模型不会重新加载;改个种子,文本编码不会重跑。试试:

3. 智能显存管理(comfy/model_management.py)

这是 ComfyUI 相比同类工具最硬核的部分:它统一调度 GPU 显存与系统内存,按需把模型权重在 CPU/GPU 之间分块换入换出(offload),支持多种精度(FP16/BF16/FP8)与量化权重,并能只加载当前阶段用到的子模型。因此 SDXL、Flux 这类大模型能在消费级低显存显卡上跑起来,而不是简单地 OOM。

4. 全程 WebSocket 反馈

执行期间,后端通过 WebSocket 向前端推送 executing(当前节点)、progress(采样进度条)、executed(节点产物,如预览图)等事件——首屏动画里节点逐个变绿,就是在模拟这条事件流。任何外部程序监听同一条 WS,也能拿到完整执行状态,这是把 ComfyUI 当纯后端用的基础。

分层架构:从浏览器到显卡

ARCHITECTURE

整个仓库可以读成自上而下的六层,层间边界清晰——前端可整体替换、节点层可无限扩展、核心运行时保持稳定。点击展开每一层:

基于 Vue 3 + TypeScript + litegraph.js 的节点画布:拖拽、连线、分组、子图、模板库、模型管理界面。前端已从主仓库拆分为独立项目,构建产物随主仓库分发;也存在第三方前端——因为它与后端之间只隔着一层 HTTP/WS 协议。桌面版(Electron 封装)与 Comfy Cloud 同样复用这一层。
用 aiohttp 实现的异步 Web 服务:POST /prompt 提交工作流、GET /history 查询结果、/upload/image、/view 取产物,外加 WebSocket 通道推送执行事件。这一层让 ComfyUI 天然是一个「生成即服务」后端。
队列管理(PromptQueue)、工作流 JSON 校验、DAG 拓扑排序、节点级缓存与增量执行、错误定位(报错精确到节点)。它不关心节点内部做什么,只负责「按依赖顺序、最小代价地把图跑完」。
所有能力的「原子单元」。内置节点(加载器、编码器、采样器、后处理等)定义在 nodes.py 与 comfy_extras/;第三方节点放进 custom_nodes/ 目录即被自动扫描注册,与内置节点完全同权——这是生态爆发的架构原因。
项目的心脏:sd.py(模型装配)、model_detection.py(自动识别权重架构)、samplers.py(采样算法族)、model_management.py(显存调度)、ldm/(UNet/DiT/VAE 等网络实现)、text_encoders/(CLIP/T5 家族)。核心库以 comfy 包形式可被独立引用。
PyTorch 抽象了硬件差异:NVIDIA CUDA、AMD ROCm、Apple Silicon MPS、Intel XPU,乃至纯 CPU 都能跑。权重文件以 safetensors 为主流格式,放置于 models/checkpoints、models/loras 等约定目录。

技术栈速览

TECH STACK

前端

  • Vue 3 + TypeScript:应用框架(独立仓库维护)
  • litegraph.js(深度定制):节点画布内核
  • WebSocket:实时进度与预览
  • Electron:官方桌面版封装

后端

  • Python 3.9+:全部服务端逻辑
  • aiohttp:异步 HTTP + WS 服务
  • JSON 工作流:图的序列化格式,可嵌入 PNG 元数据
  • comfy-cli / Manager:安装与节点包管理生态

推理

  • PyTorch:张量计算与多后端硬件支持
  • safetensors:安全权重格式
  • FP16 / BF16 / FP8 / 量化:低显存运行手段
  • xFormers / PyTorch SDPA:注意力加速

CAPABILITIES

SD1.5 / SD2 / SDXL / SD3.x / Flux / 混元 DiT / PixArt / Qwen-Image 等;文生图、图生图、局部重绘 inpaint / outpaint。

典型使用方式与二次开发

USAGE & EXTEND

四种由浅入深的打开方式——从纯用户到深度二开:

用户路径:桌面版一键安装,或克隆源码运行。模型放入 models/ 对应子目录,浏览器打开 127.0.0.1:8188,从模板库载入工作流即可出图。装上 ComfyUI Manager 后,缺失节点可一键补齐。

# 源码方式
git clone https://github.com/comfyanonymous/ComfyUI
cd ComfyUI && pip install -r requirements.txt
python main.py          # 默认 http://127.0.0.1:8188

# 或使用官方 CLI
pip install comfy-cli && comfy install

程序员路径:在界面里调好工作流,菜单导出「API Format」JSON,然后任何语言 POST 给后端。改字段 = 改参数,ComfyUI 就成了你产品里的生成微服务:

import json, urllib.request

wf = json.load(open("workflow_api.json"))
wf["6"]["inputs"]["text"] = "a cat in a spacesuit"  # 改提示词
wf["3"]["inputs"]["seed"] = 42                       # 改种子

req = urllib.request.Request(
    "http://127.0.0.1:8188/prompt",
    data=json.dumps({"prompt": wf}).encode(),
    headers={"Content-Type": "application/json"})
urllib.request.urlopen(req)   # 之后经 WebSocket / GET /history 取结果

核心二开路径:自定义节点就是一个普通 Python 类,声明输入/输出/入口函数,放进 custom_nodes/ 重启即生效——不需要动任何主仓库代码:

# custom_nodes/my_pack/__init__.py
class ImageBrightness:
    @classmethod
    def INPUT_TYPES(cls):
        return {"required": {
            "image": ("IMAGE",),                       # 类型即端口颜色
            "factor": ("FLOAT", {"default": 1.0,
                        "min": 0.0, "max": 3.0, "step": 0.05}),
        }}
    RETURN_TYPES = ("IMAGE",)
    FUNCTION = "run"
    CATEGORY = "image/adjust"

    def run(self, image, factor):   # image: torch.Tensor [B,H,W,C]
        return ((image * factor).clamp(0, 1),)

NODE_CLASS_MAPPINGS = {"ImageBrightness": ImageBrightness}
NODE_DISPLAY_NAME_MAPPINGS = {"ImageBrightness": "图像亮度"}

要点:INPUT_TYPES 决定端口与参数控件;RETURN_TYPES 决定输出端口;张量约定为 [B,H,W,C]、值域 0–1。写好后可发布到 Comfy Registry 供 Manager 安装。

前端扩展:节点包内放一个 JS 目录,通过 app.registerExtension 注入界面逻辑(右键菜单、自定义控件、画布行为):

// custom_nodes/my_pack/web/ext.js
import { app } from "../../scripts/app.js";

app.registerExtension({
  name: "my_pack.ui",
  async nodeCreated(node) {
    if (node.comfyClass === "ImageBrightness")
      node.color = "#2a4d3a";   // 给自己的节点上个色
  }
});

再往深走:替换整个前端(独立仓库 Comfy-Org/ComfyUI_frontend)、以 comfy 核心包为库嵌入自己的服务、或基于 headless 模式做集群化推理调度。

仓库地图:二开前先认路

ComfyUI/ ├── main.py # 入口:启动服务与执行循环 ├── server.py # aiohttp 路由 + WebSocket ├── execution.py # DAG 执行器与缓存 ├── nodes.py # 内置核心节点定义 ├── comfy/ # ★ 模型运行时:samplers / model_management / ldm ... ├── comfy_extras/ # 更多官方节点(按功能分文件) ├── custom_nodes/ # ★ 二开入口:放这里即自动加载 ├── models/ # checkpoints / loras / vae / controlnet ... ├── input/ · output/ # 输入素材与生成产物 └── app/ · api_server/ # 用户系统、前端托管与新版 API

带走三句话

TL;DR

1. ComfyUI 把扩散模型推理拆成强类型节点图:节点 = 计算阶段,连线 = 中间数据,图 = 可复现的工作流资产。

2. 它的引擎按 DAG 拓扑执行、只重算变化的节点,并用激进的显存调度让大模型跑在小显卡上——这是它又快又省的原因。

3. 二开有四级台阶:装节点 → 调 API → 写 custom node(一个 Python 类)→ 改前端/嵌核心库,每一级都不需要动主仓库代码。
